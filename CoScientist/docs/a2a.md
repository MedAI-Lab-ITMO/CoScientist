# A2A Mode — Agent-to-Agent Architecture

CoScientist can run as a set of independent **A2A** (Agent-to-Agent) HTTP
services instead of a single in-process agent tree. Each agent runs as its own
server; the orchestrator calls them over the [A2A protocol](https://google.github.io/A2A)
(JSON-RPC over HTTP) instead of in-process `AgentTool` calls.

The original in-process mode still works unchanged — A2A is additive. Both
modes are assembled from the same declaration: **`CoScientist/agents/system.yaml`**.

---

## 1. How it works

```
                ┌────────────────────────┐
   user ───────▶│   OrchestratorAgent    │  :8000
                │  (LlmAgent + remotes)  │
                └──┬───┬───┬───┬───┬───┬─┘
        A2A/HTTP   │   │   │   │   │   │
          ┌────────┘   │   │   │   │   └──────────┐
          ▼            ▼   ▼   ▼   ▼              ▼
      Planner   Hypotheses Research TaskExec  Medical  Coder
       :8001      :8002    :8003    :8004      :8005   :8006
```

- The whole system — agents, tools, callbacks, prompts, AND the A2A exposure —
  is declared in `agents/system.yaml` and built by `CoScientist.assembly`.
- Every agent with an `a2a:` section is wrapped as an **A2A FastAPI server** by
  `make_a2a_app()`; its **AgentCard** is generated by `make_agent_card()` from
  the same YAML (name, description, skill) and published at
  `/.well-known/agent-card.json` (and the deprecated `/.well-known/agent.json`).
- The orchestrator is built with `build_system(remote_subagents=True)`: every
  subordinate becomes a `RemoteA2aAgent` pointing at the sub-agent's card URL.
  From the orchestrator LLM's point of view they are still just tools — same
  roster, prompt and critic wiring as in-process, by construction.
- Every server attaches an **Opik tracer** so all events/callbacks land in the
  `adk-coscientist` Opik project.

### Key files

| File | Role |
|------|------|
| `agents/system.yaml` | Single source of truth: agents, tools, callbacks, prompts, HITL, A2A keys/ports/skills |
| `assembly/` | The assembler: registries, schema, prompt rendering, agent construction |
| `a2a/config.py` | Ports + URLs, derived from the YAML (override via `<KEY>_PORT` env vars) |
| `a2a/server.py` | `make_agent_card()` + `make_a2a_app()` — wraps any ADK agent as an A2A server + Opik tracing |
| `a2a/orchestrator.py` | The remote-mode orchestrator (`build_system(remote_subagents=True)`) |
| `a2a/serve.py` | Serve ONE agent by its a2a key (replaces the per-agent server modules) |
| `a2a/run_all.py` | Launches every a2a-exposed agent in one process |
| `a2a/benchmark.py` | CLI client: live event streaming + latency testing |
| `agent.py` | `adk web` entry point — `A2A_MODE=1` switches to the A2A orchestrator |

## 2. Running

All commands run from the repo root (`/app`), **not** from `CoScientist/`
(running inside the package shadows the stdlib `logging` module).

### Everything at once

```bash
python -m CoScientist.a2a.run_all
```

Starts a server for every agent with an `a2a:` section in `system.yaml`.
Ports are printed on startup.

### A single agent (for development)

```bash
python -m CoScientist.a2a.serve hypotheses    # :8002
python -m CoScientist.a2a.serve orchestrator  # :8000, sub-agents must be running
```

The argument is the agent's `a2a.key` from `system.yaml`. Pre-import env
defaults from the agent's `a2a.env` section (e.g. the coder's shared workspace
id) are applied automatically.

### The orchestrator with a web UI

```bash
A2A_MODE=1 adk web --port 8000
# sub-agents must already be running (e.g. via run_all)
```

### Configuration

Default ports live in `system.yaml` and are overridable per agent via
`<KEY>_PORT` (the upper-cased `a2a.key`):

```bash
HYPOTHESES_PORT=9002 A2A_HOST=0.0.0.0 python -m CoScientist.a2a.run_all
```

Env vars: `ORCHESTRATOR_PORT`, `PLANNER_PORT`, `HYPOTHESES_PORT`,
`RESEARCH_PORT`, `TASK_EXECUTION_PORT`, `MEDICAL_PORT`, `CODER_PORT`,
`A2A_HOST`, `A2A_DISABLE_OPIK` (set to `1` to turn off tracing).

> **Restart after edits.** A running `run_all` holds the old code (and the old
> YAML) in memory. After changing `system.yaml`, an agent, or a prompt
> template, restart the process.

## 3. Testing & observing

`a2a/benchmark.py` is a CLI client. By default it streams every internal event
(status transitions, agent thoughts, tool calls/results, artifacts) live:

```bash
# Watch everything happen, in real time:
python -m CoScientist.a2a.benchmark --agent hypotheses \
    --text "Generate a hypothesis about why sleep aids memory."

# Latency only, 5 runs:
python -m CoScientist.a2a.benchmark --agent research \
    --text "What is CRISPR?" -n 5 --no-stream
```

`--agent` accepts any `a2a.key`: `orchestrator`, `planner`, `hypotheses`,
`research`, `task_execution`, `medical`, `coder`.

Traces also appear in the **Opik dashboard** under project `adk-coscientist`.

**Per-agent inner activity.** Each server logs its agent's thoughts, tool calls,
and tool results to its own stdout (aggregated in the `run_all` console),
labeled by agent name — so when the orchestrator delegates to e.g. `CoderAgent`
you see the coder's own `execute_bash`/`check_job` calls, not just the final
result that comes back over A2A. The benchmark client only streams the agent it
hits directly; watch the `run_all` console for the full picture. The same
plugin is attached to the `App` in `agent.py`, so plain `adk web` (no A2A)
prints the same trace. The trace is also appended (ANSI stripped) to
`AGENT_LOG_FILE` (default `/app/agent_events.log`; set it to `""` for
console-only). Disable everything with `LOG_AGENT_EVENTS=0` (the older
`A2A_LOG_EVENTS=0` still works).

**Stopping the stack.** Ctrl+C (SIGINT/SIGTERM) stops all servers gracefully;
a second Ctrl+C forces an immediate exit. An in-flight request can't block
shutdown past `A2A_SHUTDOWN_TIMEOUT` seconds (default 8).

### Raw protocol check

```bash
# Agent card
curl http://localhost:8002/.well-known/agent-card.json

# Send a message
curl -X POST http://localhost:8002/ -H "Content-Type: application/json" -d '{
  "jsonrpc":"2.0","id":"1","method":"message/send",
  "params":{"message":{"messageId":"m1","role":"user",
    "parts":[{"kind":"text","text":"Hello"}]}}}'
```

## 4. Adding a new agent

> The full development guide (tools, callbacks, prompt templates, HITL,
> composites, debugging) is [development.md](development.md); below is the
> short version focused on A2A.

The `agents/` package:

```
agents/
  system.yaml          # THE system declaration — agents, tools, callbacks, prompts, A2A
  common.py            # make_llm(), make_coder_llm(), shared settings
  custom_agents.py     # custom BaseAgent classes (registered in assembly/bindings.py)
  callbacks/           # callback implementations (registered in assembly/bindings.py)
  prompts/             # templates.py (prompt templates) + builder.py
assembly/
  registry.py          # name -> object registries (tools, callbacks, prompts, ...)
  bindings.py          # registers every tool/callback/class/schema + their ToolDocs
  schema.py            # pydantic schema + loader for system.yaml
  prompting.py         # PromptContext — renders <<TOOLS>>/<<AGENTS>>/<<ROUTING>>/<<HITL>>
  assembler.py         # build_system()
```

Everything is driven by `system.yaml` — the orchestrator's roster and prompt,
the critic roster, the A2A card, ports, and `run_all` all update automatically.

### Step 1 — Register the tools (if any are new)

In `assembly/bindings.py`, add a `ToolEntry`: a factory returning the
tool/toolset and a `ToolDoc` per tool (name, signature, purpose). The docs are
what gets rendered into the agent's prompt `<<TOOLS>>` section — they are the
ONLY place tool descriptions live, so the prompt can never advertise a tool the
agent does not have. Mark MCP toolsets `runtime_resolved=True` and toolsets
that may be unconfigured `optional=True`.

### Step 2 — Add the prompt template

In `agents/prompts/templates.py`, register a template under a new name:

```python
@_register("my_agent")
def my_agent(ctx: PromptContext) -> str:
    return render_template('''
You are MyAgent...

<<TOOLS>>

## Workflow ...

<<HITL>>
''', TOOLS=ctx.render_tools(), HITL=ctx.render_hitl())
```

Never hand-write tool or agent names into the text — use the placeholders and
`ctx` helpers (`has_tool`, `has_subordinate`, `render_sibling_roster`, ...) for
anything conditional.

### Step 3 — Declare the agent in `system.yaml`

```yaml
  MyAgent:
    class: llm
    description: what this agent does (used in the parent prompt, the AgentTool
      declaration, and the A2A card — one text, three uses)
    routing: when the orchestrator should pick it
    planning: how the planner's roster describes it
    prompt: my_agent
    tools: [my_toolset]
    output_key: my_results
    hitl: false
    a2a:
      key: my_agent
      port: 8007
      skill: {id: do, name: Do, description: ..., tags: [...]}
```

and add `MyAgent` to the orchestrator's `subordinates:` list.

### Step 4 — Validate

```bash
python -m CoScientist.assembly        # config + build-order check
pytest tests/unit/test_assembly.py -q # prompt/wiring consistency invariants
```

That's it. `run_all` picks the agent up from the YAML; check it with
`python -m CoScientist.a2a.benchmark --agent my_agent --text "..."`.

> **Note on imports:** declaring agents in YAML is code organisation, not
> import-time isolation — building the system loads every agent's tools, so
> every server transitively initialises all agents at startup.

## 5. Notes & gotchas

- **AgentCard required fields:** `name`, `description`, `url`, `version`,
  `capabilities`, `defaultInputModes`, `defaultOutputModes`, `skills` —
  `make_agent_card()` fills them all from `system.yaml`.
- **`streaming=True`** must be set on the card or the server rejects
  `message/stream` requests (`make_agent_card()` always sets it).
- **HITL agents block on console input.** Agents with `hitl: true` (`planner`,
  `research`) use `request_approval`/`request_selection`, which read from
  *their server's* stdin — over A2A that means the request hangs. For automated
  testing set `HITL__ENABLED=false` in `.env`, or test non-HITL agents
  (`hypotheses`, `medical`). When HITL is off, the HITL tools AND the matching
  prompt section are both dropped automatically.
- **`task_execution`** needs the RAG DB + FEDOT.MAS reachable for its
  tool-discovery step.
- **JSON-RPC errors return HTTP 200** with the error in the body (per spec) —
  don't treat 200 as unconditional success.
- **Agent card endpoint:** prefer `/.well-known/agent-card.json`;
  `/.well-known/agent.json` is deprecated in the A2A SDK and will be removed.
